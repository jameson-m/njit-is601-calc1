Implementing pylint

A linter is a code analysis tool that is used to flag programming errors, bugs, and stylistic errors in code. Typically this is something that is used prior to code execution or compilation in order to prevent these errors or bugs from happening. For example, if a function is expecting a string for one of its parameters but an integer is given in one of the function calls, this could lead to a potentially hard to track error. A linter could catch this mistake before code execution and notify the programmer to fix it properly. There are many other things a linter can check for that are extremely helpful for a coder.

For programmers who write code in python, there is a linter called pylint that can be used for this purpose. Pylint can be configured to check for all sorts of issues including coding standards (style) and type checking. While writing python code, the programmer should do their best to try and follow stylistic best practices. For more information on where to find these, please go to the PEP 8 -- Style Guide for Python Code page. Do not be too concerned with perfectly following the style--this is where pylint will help! When you run the pylint command (demonstrated below) it will check your code and warn you if you are not perfectly following the style guide. You can then easily fix these mistakes based on the given warning messages.

One really useful tool to use when using pylint is variable typing. This means that when you declare a variable you also specify what its data type should be. This allows pylint to check and make sure that variables are assigned data values of the correct type. Let’s say for example that you have a function that adds two integers. What if you accidentally call that function and pass two strings instead? This would throw an error when the application runs. By using typing and pylint, you can catch these types of mistakes before the code executes.

So, how do you use the typing functionality? Just like this:

greeting: str = "hello, world!"

In this code we are creating a variable named “greeting” and assigning it a value of “hello, world!”. After the variable is given a name there is a colon followed by the desired data type. For more information on typing in python, please see this page.

In order to run pylint, you need to run the following command within the project directory:

pylint .

This will run the pylint linter and check all files in the given directory. Additionally, you can pass an option to pytest that runs pylint as part of the testing:

pytest --pylint --cov

Glossary of OOP Terms

  • Factory
    A function that creates an instance of a class. The actual implementation of how the object is created is abstracted away from the programmer. This allows for changes to be easily made to how an object is instantiated.
  • Instance
    An object created from a class.
  • Object
    An instance of a class.
  • Class
    A blueprint for an object; contains all attributes and functions of the object.
  • Namespace
    A way to ensure variable names referring to objects are unique. For Example, if you have two variables named “greeting”, without using namespaces it would not be possible for these two variables to be named the same. Within each namespace a variable name is guaranteed to be unique.
  • Constructor
    A special type of method that is called to create an object of a given type. For example, for a class Person, a constructor can be called to create a new person with a name, age, etc.
  • Fixture
    A simple function called between tests to set up a test to run properly. For example, in a calculator program it would be important to clear the calculator between tests so that each test runs on a reliably clean environment.
  • Type Hint
    Hints given in an IDE to help the programmer know what type a variable should be.
  • Type Cast
    Conversion between data types; for example, a string “23” can be type cast to an int 23.
  • Unit Test
    Automated tests written by a software developer to test a small section of an application. These tests might test a single function or a single item within a function. For example, let’s say you have a divide function. You might have one test that tests dividing two numbers produces the desired result and another test that checks that dividing by zero errors are handled properly. Each of these tests is a unit test.
  • Static
    A variable marked as “static” is allocated in memory for the duration of the program. Classes with static methods do not need to be instantiated before these methods are used.
  • Class Method
    Methods for a specific class. For example, a Car class would have a “honk” method but a Airplane class would not. The “honk” method is a class method.
  • Facade
    The facade pattern is an object that serves as a front-facing interface masking more complex underlying code.
  • Method
    A function that is part of any objects of a given class.
  • Property
    A variable specific to objects of a given class. For example, a Car class would have properties such as color or number of doors.
  • Abstraction
    Hiding internal details of a given object from the user.
  • Encapsulation
    Combining an object’s attributes and functionality into a single package (class); allows the programmer to access properties and methods of a class without needing to worry about how a class is implemented.
  • Inheritance
    Create a new class from an existing class. The new class inherits properties and methods from the base class.
  • Base Class
    Original or parent class from which dervied classes inherit properties and methods.
  • Derived Class
    Newly created class that inherits properties and methods from the base class. Additional new properties and methods can be created specifically to this new class.
